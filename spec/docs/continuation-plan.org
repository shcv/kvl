#+TITLE: Continuation/Multiline Semantics Plan

Detailed plan for auditing and fixing multiline value handling across all implementations.

* The Rule

Indentation beyond the parent key is continuation. The continuation block is
then recursively parsed for KVL content:

1. *Capture*: all lines indented deeper than the parent are part of the block
2. *Re-parse*: check if the block is "proper" KVL:
   - Every line at the block's base indentation must have a separator
   - Lines indented deeper than base are sub-continuations of the preceding key
   - If ALL base-level lines have separators → nested KVL structure
   - If ANY base-level line lacks a separator → entire block is plain text
3. *Whitespace*:
   - =parse()= (low-level): preserves raw whitespace including leading =\n= and absolute indentation
   - =loads()= (high-level): strips leading =\n=, dedents to the block's base indentation level

* Examples

#+begin_src kvl
/= Case 1: Nested KVL (all base lines have separator)
server =
    host = localhost
    port = 8080
/= → {server: {host: "localhost", port: "8080"}}

/= Case 2: Plain text (no base line has separator)
description =
    This is a long
    description text
/= → {description: "This is a long\ndescription text"}

/= Case 3: Plain text (mixed — one base line lacks separator)
notes =
    line one
    key = value
/= → {notes: "line one\nkey = value"}

/= Case 4: Key with inline value AND indented children
/= The inline value is the first entry; children follow
server = primary
    host = localhost
    port = 8080
/= → {server: {primary: {}, host: "localhost", port: "8080"}}
/= (inline value "primary" becomes a categorical entry)

/= Case 5: Nested continuation (sub-key has its own multiline value)
config =
    greeting =
        Hello there,
        welcome aboard
    port = 8080
/= → {config: {greeting: "Hello there,\nwelcome aboard", port: "8080"}}

/= Case 6: Deeply nested
a =
    b =
        c =
            d = value
/= → {a: {b: {c: {d: "value"}}}}
#+end_src

* Known Issues

** Go: deeper indent after valued key
- Currently treats =server = primary\n    host = localhost= as multiline
  continuation ("primary\n    host = localhost") instead of nested structure
- Should check if the continuation block is proper KVL and nest if so
- Fix: apply the re-parse rule to blocks following valued keys, same as empty-value keys

** Spec wording
- Current multiline section in language-specification.org uses "two-stage processing" language
- Should be tightened: "continuation block is recursively parsed"
- Clarify that base-level means "relative to the continuation's indentation, not the file"
- Add Case 4 (valued key with children) to spec examples
