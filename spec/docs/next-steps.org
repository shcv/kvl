#+TITLE: KVL Next Steps

Decisions and planned work from the deep review (2026-02-27).

* Decided: Spec Clarifications

** Escape sequences: simple rule, no pair processing
- =\{sep}= → ={sep}=. That's the only escape.
- All other =\= are literal characters.
- No pair processing, no doubling, no recursion.
- Separator detection: if =\= is immediately before separator, it's escaped. Period.
- Consequence: impossible to have a key ending in =\= with zero whitespace before the separator, but whitespace separation makes this unnecessary in practice.

** Inline separator splitting: first occurrence only
- =a = b = c= → key =a=, value =b = c=
- Matches CCL behavior (confirmed from OCaml source and test suite)

** No type inference in core spec
- All values are strings at the parse level
- Type inference is application-layer, not part of =loads()=
- Schemas can declare types when validation is needed

** Empty files parse to ={}=
- Zero statements is valid per the grammar
- =parse("")= and =loads("")= both return empty dict/object

** Indentation levels are dynamic
- Each level's indent is determined by its first child relative to its parent
- No fixed indent unit — 2 spaces, 4 spaces, tabs all valid
- Only constraint: children must be indented more than parent, siblings must match

** Ordering and duplicates
- Categorical merge is set-based: =A + B = B + A= as sets
- Ordering in output is implementation-defined (use OrderedDict for determinism)
- Duplicates are inherently lost: =tags = red; tags = red= → ={tags: {red: {}}}=
- Applications needing ordered lists or duplicates should use numbered keys

** Multiline value semantics
- Indentation beyond the parent key = continuation
- Low-level (=parse()=): preserves raw whitespace (leading =\n=, absolute indentation)
- High-level (=loads()=): strips leading =\n=, dedents to the level of the second line
- Continuation block is recursively parsed for KVL content:
  - All lines at the block's base indentation must have a separator → nested KVL
  - Any line at base indentation without separator → entire block is plain text
  - Only relatively indented lines (deeper than block base) can be sub-continuations
- This is all-or-nothing: mixed blocks are always plain text

** Comment handling
- =/== comments leak into parsed data by design (preserves comments for round-trip)
- Higher-level interfaces should support stripping comments before returning
- Mechanism: comment key declaration in header + argument to =loads()= + utility function
- Raw =parse()= always preserves comments

* Decided: New Features

** Comment stripping API
- Header option to declare comment keys (default: =/==)
- =loads()= parameter to strip comments before returning
- Standalone utility function to strip comments from parsed data
- Available across all implementations

** Merge policy helpers
- Helper functions for common policies: first-wins, last-wins, choice
- Arguments to =compact()= or separate utility functions
- Categorical merge is correct foundation; policies are app-level convenience

** Numbered keys to arrays (schema level)
- Schema can declare =items = array(start=0)= or =array(start=1)=
- Numbered keys compact to arrays at schema validation level
- Natural extension of schema system, no new syntax needed
