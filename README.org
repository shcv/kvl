#+TITLE: KVL - Key-Value Language
#+SUBTITLE: Small configuration files that merge cleanly

* Overview

KVL is a small, indentation-based configuration format for settings that need
to be read by people, loaded by programs, layered across environments, and
validated before use.

The everyday interface is intentionally plain:

- =load= and =loads= parse KVL into normal dictionaries or objects.
- Python, JavaScript, Go, and Zig all expose serializers that can write normal
  dictionaries or objects back to KVL.
- Repeated keys become lists in the public API.
- Schema validation is optional and can convert strings into typed values.
- The lower-level representation is available for implementers, but most users
  should not need it.

* Example

#+begin_src kvl
/= Comments start with /=

name = checkout
environment = production

server =
  host = app.example.com
  port = 8080

features =
  search = true
  recommendations = false

tags = web
tags = api
tags = public
#+end_src

In Python:

#+begin_src python
import kvl

config = kvl.loads("""
name = checkout
server =
  host = app.example.com
  port = 8080
tags = web
tags = api
""")

assert config["server"]["host"] == "app.example.com"
assert config["tags"] == ["web", "api"]
#+end_src

KVL leaves scalar values as strings by default. Use a schema when an application
needs typed values such as integers, booleans, dates, or constrained strings.

* Syntax

KVL has three main forms.

** Key-Value Pairs

#+begin_src kvl
name = checkout
port = 8080
debug = false
#+end_src

Values do not need quotes. The parser preserves them as strings unless schema
validation converts them.

** Nested Objects

#+begin_src kvl
server =
  host = localhost
  port = 8080
#+end_src

Indentation creates nested data. A key with no value introduces the nested
block below it.

** Lists

The public API treats repeated keys as lists:

#+begin_src kvl
tags = web
tags = api
tags = production
#+end_src

#+begin_src json
{
  "tags": ["web", "api", "production"]
}
#+end_src

KVL also supports header-declared list markers when that reads better for long
lists:

#+begin_src kvl
#= kvl 1.0 -

dependencies =
  - postgresql
  - redis
  - s3
#+end_src

List markers also give KVL a clean nested-list and list-of-object surface:

#+begin_src kvl
#= kvl 1.0 -

groups =
  -
    - a
    - b
  -
    - c

servers =
  -
    name = web1
    port = 80
  -
    name = web2
    port = 81
#+end_src

The parser also accepts a compact shorthand where the first nested entry stays
on the list-item line, such as =- - a= or =- name = web1= with additional
indented child lines. Serializers currently normalize that shorthand back to
the bare-marker form shown above.

All four current language implementations can round-trip that list-marker
surface. Python and JavaScript expose the most ergonomic serializer switches;
Go and Zig use lower-level config or helper APIs for the same nested-list and
object-list forms.

* Configuration Merging

KVL is designed for layered configuration: defaults, deployment settings,
environment overrides, and local files can be combined in a predictable way.

#+begin_src kvl
/= base.kvl
service =
  host = localhost
  port = 8080

tags = web
tags = api
#+end_src

#+begin_src kvl
/= production.kvl
service =
  host = app.example.com
  tls = true

tags = production
#+end_src

Merging those files preserves shared structure and adds new information:

#+begin_src json
{
  "service": {
    "host": ["localhost", "app.example.com"],
    "port": "8080",
    "tls": "true"
  },
  "tags": ["web", "api", "production"]
}
#+end_src

This is useful when configuration layers should compose without silently losing
values. Applications that want single-winner override behavior can apply that
policy after parsing or schema validation; KVL keeps the merge step lossless.

* Schemas

Schemas are optional. They validate fields and can convert KVL strings into
application types.

#+begin_src kvl
#= kvl 1.0 schema open optional

port =
  type = int
  min = 1
  max = 65535

tls = bool
tags = list
#+end_src

Python schema usage:

#+begin_src python
from kvl import Schema

schema = Schema.from_file("server.schema.kvl")
config = schema.load("server.kvl")

assert isinstance(config["service"]["port"], int)
#+end_src

See [[file:spec/docs/schema-guide.org][the schema guide]] for the full validation model.

* Command Line

The Python package provides the most complete CLI today:

#+begin_src shell
kvl validate config.kvl
kvl validate config.kvl --schema config.schema.kvl
kvl validate config.kvl --schema config.schema.kvl --json
kvl schema-convert config.kvl config.schema.kvl --json
kvl format config.kvl
kvl merge base.kvl production.kvl local.kvl
kvl convert config.kvl config.json
#+end_src

The schema-aware CLI commands accept =--json= to emit stable machine-readable
status objects. The result envelope includes the input file, schema file, and
structured error details such as the validation stage and field path when
available.

Other implementations also expose parser or conversion CLIs for conformance
testing and language-specific use. Public serializer options such as direct
scalar rendering, explicit =null= text for =None=, and list-marker selection are
currently exposed most clearly through the library APIs.
